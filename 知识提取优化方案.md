# 栩格 知识提取「缺斤少两」根因分析与优化方案

> **目标：** 解决上传 PDF / Word / 文字笔记后，AI 提取的知识点内容不全、遗漏严重的问题。
> **适用范围：** `server/src/` 下的文件解析、文本提取、AI 分析链路。
> **供 Trae 参考修改。**

---

## 一、链路全景回顾

```
文件上传 (multer)
    ↓
extractContent.ts ────→ PDF: pdf-parse (主) / pdf-lib (降级)
                      └── DOCX: mammoth → 转纯文本
    ↓
extractContent.ts ────→ 图片提取 (PDF XObject / DOCX word/media/)
    ↓
ai.ts → analyzeNote() ──→ 图片分析 (LLM Vision)
                         └── 文本 + 图片分析结果 → 截取前15000字符 → LLM 提取知识点
    ↓
notes.ts → 异步保存知识点到数据库
```

### 问题 1：LLM 输入截断（最严重）

**影响文件：** `server/src/services/ai.ts`

**现状：** 第 98 行 `fullContent.slice(0, 15000)` 将传入 LLM 的内容硬截断为最多 15000 个字符（约 7500 中文字）。
对于篇幅稍长的笔记，后半部分完全丢失，LLM 无从提取知识点。

```
// 当前代码
${fullContent.slice(0, 15000)}
```

**根因：** 设计时为防 token 超限，但 DeepSeek-chat 上下文窗口为 64K tokens（约 8-10 万中文字），
当前截断只用了不到 20% 的容量，严重浪费。

**解决方案：**

```typescript
// 方案 A：直接扩大截断上限（最简单、效果最显著）
// 将 15000 改为 80000，一次覆盖绝大多数文档
${fullContent.slice(0, 80000)}

// 方案 B：根据实际 token 数动态截断（更精确）
// 引入简单的 token 估算（中文约 1.5 字符/token），保留余量
const MAX_TOKENS = 55000; // 保留 9000 tokens 给输出
const estimatedTokens = fullContent.length / 1.5;
const safeContent = estimatedTokens > MAX_TOKENS
  ? fullContent.slice(0, Math.floor(MAX_TOKENS * 1.5))
  : fullContent;
```

**预期效果：** 大部分文档可完整输入，消除 60-70% 的「缺斤少两」。

### 问题 2：PDF 文本提取降级路线产出垃圾数据

**影响文件：** `server/src/utils/extractContent.ts`

**函数：** `extractPdfTextWithPdfLib()`

**现状：**

```typescript
async function extractPdfTextWithPdfLib(buffer: Buffer): Promise<{ text: string }> {
  // ...
  const contentStream = node?.Contents?.();
  if (contentStream) {
    const str = contentStream.asString ? contentStream.asString() : '';
    text += str + '\n';
  }
  // ...
}
```

`contentStream.asString()` 读取的是 PDF 底层页面内容流操作码（如 `(Hello) Tj (World) Tj` 混合坐标变换指令），
输出的不是人类可读的连续文本，而是夹杂大量 PDF 指令的二进制碎片。
当 pdf-parse 失败时走这条降级路线，实际产出了垃圾数据喂给 LLM，导致提取结果混乱。

**解决方案：**

```typescript
// 方案：去掉 pdf-lib 假降级，改用 pdfjs-dist
// 安装: npm install pdfjs-dist
import * as pdfjsLib from 'pdfjs-dist';

async function extractPdfTextWithPdfJs(buffer: Buffer): Promise<{ text: string }> {
  try {
    const loadingTask = pdfjsLib.getDocument({ data: buffer.buffer });
    const pdf = await loadingTask.promise;
    let fullText = '';
    for (let i = 1; i <= pdf.numPages; i++) {
      const page = await pdf.getPage(i);
      const content = await page.getTextContent();
      // 利用 items 的 x/y 坐标判断换行与段落
      let lastY = -1;
      for (const item of content.items) {
        if ('str' in item) {
          if (lastY >= 0 && Math.abs(item.transform[5] - lastY) > 5) {
            fullText += '\n';
          }
          fullText += item.str;
          lastY = item.transform[5];
        }
      }
      fullText += '\n\n';
    }
    return { text: fullText.trim() };
  } catch (err) {
    console.error('[Extract] pdfjs-dist failed:', err);
    return { text: '' };
  }
}
```

**或者简化方案（不改依赖）：** pdf-parse 失败时不要走 pdf-lib 降级，直接返回空文本并标记为"可能为扫描版需 OCR"，
避免垃圾数据污染 LLM。

```typescript
// 简化降级（不改依赖）
async function extractPdfTextFallback(buffer: Buffer): Promise<{ text: string }> {
  // pdf-lib 降级不可用，直接返回空
  return { text: '' };
}
```

### 问题 3：PDF 图片提取遗漏 FlateDecode 编码图片

**影响文件：** `server/src/utils/extractContent.ts`

**函数：** `extractPdfImages()`

**现状：** 只提取了 DCTDecode（JPEG 编码）的图片，FlateDecode 编码（含 PNG 图片）被直接跳过。

```typescript
if (filter === PDFName.of('DCTDecode')) {
  // JPEG 格式，数据已经是标准 JPEG
  rawData = obj.getContents ? obj.getContents() : obj.contents;
} else if (filter === PDFName.of('FlateDecode')) {
  continue;  // ← 直接跳过！大量内嵌图表被遗漏
}
```

PDF 中嵌入的示意图、流程图、公式截图经常使用 FlateDecode（PNG），
跳过这些意味着**文档中的视觉知识内容（图示、表格截图、公式截图）几乎没有被提取**。

**解决方案：**

```typescript
import { inflate } from 'zlib';
import { promisify } from 'util';
const inflateAsync = promisify(inflate);

// 在 extractPdfImages 的 filter 判断中追加 FlateDecode 处理：
else if (filter === PDFName.of('FlateDecode')) {
  // FlateDecode 可能是 PNG 或原始像素数据
  try {
    const rawCompressed = obj.getContents ? obj.getContents() : obj.contents;
    if (!rawCompressed) continue;
    
    const decompressed = await inflateAsync(Buffer.from(rawCompressed));
    
    // 检测是否为 PNG 格式（文件头: 89 50 4E 47 0D 0A 1A 0A）
    const pngSignature = Buffer.from([0x89, 0x50, 0x4E, 0x47, 0x0D, 0x0A, 0x1A, 0x0A]);
    if (decompressed.slice(0, 8).equals(pngSignature)) {
      rawData = decompressed;
    }
    // 否则是原始像素数据（不带文件头），跳过或做 BMP 重建
  } catch {
    continue;
  }
}
```

**同时调整图片数量上限：**

```typescript
// 将 5 张改为 20 张，足以覆盖大部分文档中的重要图示
return images.slice(0, 20);
```

### 问题 4：图片分析和文本分析互相挤压 token 预算

**影响文件：** `server/src/services/ai.ts`

**现状：** 流程是先分析图片得到文本描述，再把文本描述和文档原文拼接，然后一起截断 15000 字符。
图片分析结果越长，文本原文被截得越短，两者互相竞争有限的空间。

```
// 当前逻辑
imageAnalysis = await analyzeImages(...) // → 可能输出3000字描述
fullContent = `[文档文本内容]\n${content}\n\n[文档图片内容分析]\n${imageAnalysis}`
// → 然后 .slice(0, 15000) → 图片描述占了3000，文本只剩12000
```

**解决方案 A（推荐）：直接使用多模态能力**

如果模型支持多模态（DeepSeek 当前版本不支持图片输入），
将图片作为 base64 直接传入 LLM，避免"图片→文字转述"的信息损耗。

**解决方案 B（实用方案）：分离文本分析和图片分析**

```typescript
// 分别分析，结果单独存储
const textResult = await analyzeTextContent(content);  // 分析纯文本
const imageResult = images.length > 0
  ? await analyzeImageContent(images)                   // 分析图片
  : null;

// 合并时先放文本结果，图片结果作为补充附加到 summary 中
// 文本分析用满 token 预算，图片分析独立调用
```

### 问题 5：扫描版 PDF 检测阈值过于粗糙

**影响文件：** `server/src/utils/extractContent.ts`

**现状：**

```typescript
const isScannedPdf = text.length < 200;
```

一篇 PDF 如果只有标题 + 一两行正文（如获奖证书、简短摘要），字符数可能不到 100，
会被错误标记为扫描版 → AI 分析直接返回"该文档为扫描版 PDF，暂无法自动提取"。

**解决方案：**

```typescript
// 更精确的扫描版判断：综合文本长度和文本与文件大小的比例
// 如果文本极少但文件很大（>500KB），很可能是扫描版
// 如果文件小文本也少，可能是短文档
const fileSize = buffer.length;
const isScannedPdf = text.length < 50 && fileSize > 500 * 1024;

// 补充：检测 PDF 是否包含文本对象（有文字对象的 PDF 不是扫描版）
```

### 问题 6：LLM 的单次提取模式不适合超长文档

**影响文件：** `server/src/services/ai.ts`

**现状：** 无论文档多长，都是一次性调用 LLM 提取知识点。对于教材、手册类文档（100+页），
就算扩大截断上限，一次提取也会因"Lost in the Middle"效应（模型对中间部分记忆最差）遗漏大量知识点。

**解决方案：Map-Reduce 提取模式**

```typescript
// Map 阶段：将文档分段，每段独立提取知识点
async function analyzeNote(content: string, ...): Promise<...> {
  // 如果内容较长（>40000 字符），采用 Map-Reduce 策略
  if (content.length > 40000) {
    return analyzeNoteMapReduce(content, images, isScannedPdf);
  }
  // 否则正常单次提取
  return analyzeNoteSingle(content, images, isScannedPdf);
}

async function analyzeNoteMapReduce(content: string, ...): Promise<...> {
  // 1. 按章节/段落拆分（保留标题结构）
  const chunks = splitByChapters(content, 15000);
  
  // 2. Map：每个 chunk 独立提取知识点
  const chunkResults = await Promise.all(
    chunks.map(chunk => extractKnowledgeFromChunk(chunk))
  );
  
  // 3. Reduce：合并知识点（去重、归类）、合并摘要
  return mergeKnowledgeResults(chunkResults);
}

function splitByChapters(content: string, maxChunkSize: number): string[] {
  // 按 ## / ### / 章节标题 或双重换行分割
  // 确保每个 chunk 从完整段落/章节开始
  const chapters = content.split(/\n#{1,3}\s+/);
  // 合并太小的章节，拆分太大的章节
  return smartChunk(chapters, maxChunkSize);
}
```

### 问题 7：启发式降级方案太弱

**影响文件：** `server/src/services/ai.ts`

**函数：** `heuristicAnalyze()`、`heuristicGenerateQuestions()`

**现状：**

1. **知识点提取：** 只靠正则匹配 `/[-*•]|\\d+[.．、]/` 找列表项，对非列表型笔记（连续段落叙述）几乎失效
2. **领域猜测：** `guessDomain()` 仅靠关键词匹配（"经济"→经济学，"算法"→计算机），多学科交叉笔记只会归到第一个
3. **题目干扰项：** 固定 16 个词条的池子（机会成本、二叉树遍历、SWOT 分析……），与文档内容完全无关时毫无意义
4. **去重：** `similarity()` 基于字符集合的交并比，对中文文本区分度低

**解决方案：**

```typescript
// 1. 用 TF-IDF 或 TextRank 提取关键短语替代正则匹配
// 简单实现：基于词频-位置加权
function extractKeyPhrases(text: string, maxCount: number): string[] {
  // 分句 → 计算每个句子的 TF（词频）+ 位置权重（越靠前权重越高）
  // 选择得分最高的句子作为知识点描述
  // 从描述中提取简短名词短语作为知识点名称
}

// 2. 改进领域猜测：多标签匹配
function guessDomains(text: string): string[] {
  const scores = new Map<string, number>();
  for (const [domain, keywords] of Object.entries(domainKeywords)) {
    const count = keywords.filter(k => text.includes(k)).length;
    if (count > 0) scores.set(domain, count);
  }
  // 返回得分 > 0 的所有领域，按得分排序
  return [...scores.entries()]
    .sort((a, b) => b[1] - a[1])
    .slice(0, 3)
    .map(([d]) => d);
}

// 3. 干扰项从同一批次的知识点中交叉抽取
function generateDistractors(correctName: string, allKnowledgePoints: string[]): string[] {
  return allKnowledgePoints
    .filter(name => name !== correctName)
    .sort(() => Math.random() - 0.5)
    .slice(0, 3);
}
```

### 问题 8：异步分析缺少截断日志，问题定位困难

**影响文件：** `server/src/services/ai.ts`、`server/src/routes/notes.ts`

**现状：** `analyzeNote()` 内部截断了内容但没有记录日志，
当用户发现知识点不全时，无法判断是"截断导致的"还是"LLM提取不完整"。

// 在 ai.ts 中截断时添加日志
const truncatedLength = fullContent.length;
const actualInput = fullContent.slice(0, 80000);

  console.log(`[AI] Content truncated: ${truncatedLength} → 80000 chars (lost ${truncatedLength - 80000} chars)`);
}

// 在 notes.ts 中完成分析后记录统计

## 三、优化优先级总结

| 优先级 | 问题 | 优化内容 | 影响面 | 文件 | 改动量 |
|:------:|:----:|----------|:------:|:----:|:------:|
| **P0** | LLM 输入截断 | 15K→60K~80K 扩大上限 | 消除 60-70% 遗漏 | `ai.ts` L98 | 1行 |
| **P0** | PDF 降级出垃圾 | 去掉 pdf-lib 假降级或用 pdfjs-dist | 消除降级时的错误数据 | `extractContent.ts` | 中 |
| **P1** | 图片提取漏 FlateDecode | 增加 FlateDecode 解压+PNG检测 | 恢复大量内嵌图表 | `extractContent.ts` | 10行 |
| **P1** | 图片数量上限过低 | 5→20 张 | 长文档图片不遗漏 | `extractContent.ts` | 1行 |

| **P2** | 超长文档 single-shot | 加 Map-Reduce 分段提取 | 书籍级文档完整提取 | `ai.ts` | 大 |

## 五、验收标准

完成优化后，上传以下三种测试文档进行验证：

1. **长文档**（>20 页 PDF，如《Servlet详解.pdf》，已在上传目录中）
   - ✅ 提取的知识点数量 ≥ 人工标注的 70%
   - ✅ 后半部分的内容也被提取（不局限在前几页）

2. **含表格/图表的文档**
   - ✅ 表格数据在知识点描述中被正确引用
   - ✅ 内嵌图表的说明文字出现在分析结果中

3. **短文档**（含标题+2-3段正文）
   - ✅ 不会被误判为扫描版
   - ✅ 知识点提取完整
